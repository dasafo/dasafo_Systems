El ponente, Alan Buscalia, tiene montada una arquitectura muy avanzada basada en un **Orquestador** central y múltiples **Subagentes**, diseñada para mantener limpia la ventana de contexto y ser escalable en proyectos grandes. 

La configuración de sus agentes funciona de la siguiente manera:

*   **El Orquestador (El Capataz):** El agente principal actúa exclusivamente como un orquestador que manda y delega tareas. Este agente no lee archivos ni escribe código directamente, lo cual es clave para evitar que su ventana de memoria se llene de "ruido" y pierda contexto.
*   **Fases de Spec Driven Development (SDD):** El sistema divide el trabajo de la IA en procesos muy bien definidos, como inicialización, exploración del código, propuesta, creación de especificaciones, diseño, implementación de tareas, verificación y resumen. 
*   **Subagentes en Paralelo:** El orquestador lanza subagentes para que ejecuten las distintas fases de SDD. Cada subagente abre una sesión nueva y limpia de contexto. Además, pueden trabajar de forma asíncrona y en paralelo (por ejemplo, investigando dos cosas a la vez o haciendo el diseño y las especificaciones al mismo tiempo) para ahorrar tiempo.
*   **Habilidades dinámicas (*Skills*):** Para evitar que la IA tenga que leer todo un repositorio inmenso, el proyecto se divide en pequeñas *skills* (habilidades específicas). Los subagentes son lo suficientemente inteligentes como para leer un registro de las *skills* disponibles y cargarlas únicamente cuando las necesitan (*lazy loading*).
*   **Memoria persistente con "Engram":** Para que los agentes se comuniquen entre sí y recuerden decisiones pasadas, utilizan una herramienta propia llamada **Engram**. Es una pequeña base de datos local que indexa y guarda automáticamente el contexto, decisiones técnicas, requerimientos y patrones aprendidos del proyecto. Cuando un nuevo subagente entra en acción, primero lee Engram para empaparse del contexto al instante.
*   **Enfoque Multimodelo:** El sistema no utiliza un único modelo costoso para todo el flujo. Dependiendo de la fase, el orquestador elige el modelo más adecuado: usa **Gemini** para investigar, pensar y diseñar arquitecturas, **Sonnet** para codificar rápidamente, o **Opus** para tareas muy complejas que requieren gran razonamiento.
